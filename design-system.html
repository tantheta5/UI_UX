Chapter 04

Brand Values

The eight principles that shape every decision — from a single button label to the architecture of a multi-step flow. Values are not slogans; they are the test we apply when we don't know what to do.

Consistency

Same tokens, same rhythm, same motion — everywhere. Familiarity frees the user to focus on their work, not the interface.

Clarity

Every element should mean something. If it doesn't communicate, it doesn't belong. Whitespace is content.

Human-first

We design for people, not personas. Real fatigue, real attention spans, real joy in a small delight done well.

Trust

Predictable behavior, honest states, reversible actions. Trust is earned one interaction at a time and lost in one.

Innovation

We ship the future, calmly. Innovation isn't novelty for its own sake — it's the quiet surprise that makes the old way feel obsolete.

Accessibility

If it isn't accessible, it isn't done. We design for the widest range of abilities, contexts, and devices from the first sketch.

Inclusivity

Names, pronouns, locales, color vision, inputs — we build for the full human spectrum, not the median user.

Efficiency

Fewer clicks, fewer fields, fewer decisions. The best interface is the one that gets out of the way the fastest.

Chapter 05

Design Principles

Eight principles, drawn from the Apple Human Interface Guidelines and refined for Lumen's editorial temperament. These are the rules we return to whenever a decision is hard — the things we hold ourselves to even when no one is looking.

01
Clarity

Every element on the screen should communicate its purpose instantly. Legibility is non-negotiable. Icons accompany labels. Color reinforces meaning rather than carrying it alone. When in doubt, choose the option a tired user at 11pm can still understand.

DoPair every icon with a text label, especially for primary actions.
Don'tUse an ambiguous icon alone for navigation — users will guess wrong.
DoUse generous whitespace to let content breathe and groups be obvious.
02
Deference

The interface serves the content, never the other way around. Chrome recedes; the user's work takes center stage. We avoid heavy borders, loud backgrounds, and decorative elements that compete with the substance of the page. When the UI is doing its job, it nearly disappears.

DoLet content fill the canvas. Use subtle dividers, not heavy frames.
Don'tAdd gradients, shadows, or motion that draw attention away from content.
03
Depth

Hierarchy is communicated through layering — soft shadows, subtle blurs, and thoughtful z-axis ordering. Depth conveys meaning: a raised card is interactive; a recessed field accepts input. We never use depth decoratively; every elevation token maps to a state.

DoReserve the strongest shadows for interactive surfaces and modals.
Don'tApply drop shadows to body text or static labels.
04
Consistency

Patterns learned in one part of the system apply everywhere. A primary button looks the same in a hero, a modal, and a settings panel. This isn't aesthetic conformity for its own sake — it's how we reduce cognitive load and let users build intuition that transfers.

DoReuse the same component across every surface, themed by token.
Don'tBuild a one-off "special" button for marketing — tokenize the variant instead.
05
Feedback

Every action receives an immediate, perceptible response. Buttons compress on press, links highlight on hover, submissions show progress. The system is always telling the user "I heard you" — through motion, color, or copy. Silence breeds distrust.

DoConfirm received actions with motion, sound (optional), or copy within 100ms.
Don'tLeave a button in its default state after click — users will click again.
06
Hierarchy

Visual weight tells the user where to look first, second, third. One primary action per region. One headline per page. One accent color for emphasis. When everything is emphasized, nothing is — so we choose ruthlessly and let the rest recede.

DoDesign each view around a single primary action and a single hero metric.
Don'tFill a toolbar with three "primary" buttons of equal weight.
07
Focus

Attention is the scarcest resource. We design for a single primary task per screen and decline to surface competing CTAs. When the user is filling a form, that form is the only thing that matters — no sidebar promotions, no notification toasts.

DoHide unrelated navigation during focused flows like checkout or onboarding.
Don'tSurface upsells inside a destructive-action modal.
08
Motion

Motion explains, never decorates. It shows where things came from, where they're going, and how they relate. Durations are short and easings are gentle — springy enough to feel alive, restrained enough to never feel busy. Motion that can't be felt is wasted; motion that demands attention is worse.

DoAnimate position changes to show origin and destination.
Don'tAdd entrance animations to every section — they become noise.

Chapter 07

Content Guidelines

Words are part of the design. The wrong label can undo an hour of thoughtful UI work; the right one can make a complex flow feel inevitable. These guidelines keep Lumen's voice consistent across every surface.

Voice & Tone
Our voice is steady and warm — like a trusted editor who has seen it all. We write to be understood the first time, not to impress. Tone flexes with context: matter-of-fact in errors, celebratory in success, restrained everywhere else.
Clear

Short sentences. Active voice. One idea per paragraph. No jargon when plain language works.

Warm

Encourage, don't shame. "Add a backup email" beats "Email is required". Address the user as "you".

Honest

Tell the truth about what happened and what to do next. Never blame the user for system failures.

Grammar & Mechanics
Consistency in the small things builds trust in the large ones.
  • Sentence case for headings and buttons — never Title Case
  • Oxford comma: yes, always
  • Numbers under 10 spelled out; 10+ as digits
  • Percent: "12%" not "twelve percent"
  • Time: "3:00 PM" with a space before the meridiem
  • Dates: "Aug 14, 2025" — never "08/14/25"
  • Plurals: "1 project" · "2 projects" — never "project(s)"
  • Ellipses: only for truncated text, not for suspense
  • Em dashes for asides — like this — not parentheses
  • Avoid exclamation marks except in genuine celebration
Buttons & Labels
Button labels are verbs. They tell the user what will happen, in two or three words. Avoid "Submit", "OK", and "Click here" — they describe the action, not the outcome.
UseSave changes · Create project · Send invitation · Delete account
AvoidSubmit · OK · Click here · Do it · Proceed
Forms
Form copy is patient. Labels describe what's needed; helper text explains why; errors explain what went wrong and how to fix it. Never just say "Invalid".
Use Label: Email Helper: We'll only use this to send you receipts. Error: That email looks incomplete — did you mean jane@studio.com?
Avoid Label: Email Address * Helper: (none) Error: Invalid input.
Tooltips · Errors · Notifications
Tooltips clarify; errors explain; notifications inform. Each has a different register — keep them distinct.
Use Tooltip: Saves automatically every 30 seconds. Error: Couldn't reach the server. Your work is saved locally — we'll retry shortly. Notification: Marc left a comment on "Atlas migration".
Avoid Tooltip: Click here to save your changes. Error: Error 500: Internal server error. Notification: You have 1 new notification.
Empty States & Microcopy
Empty states are an onboarding moment, not a dead end. Microcopy is the small text that holds the user's hand — labels under icons, helper text under inputs, hints in tooltips.
Use Empty: No projects yet. Create your first to start collaborating. Microcopy: Drag a file here, or browse.
Avoid Empty: No data available. Microcopy: Please select a file from your local filesystem to upload.
Success & Warnings
Success messages are short and specific. Warnings explain the consequence of inaction.
Use Success: Saved. Your preferences are live. Warning: Your trial ends in 3 days. Add a payment method to keep your projects.
Avoid Success: Operation completed successfully! Warning: Warning: Trial expiring soon.
Chapter 08

Accessibility Guidelines

Accessibility is not a checklist at the end — it's a baseline at the start. Lumen targets WCAG 2.2 AA across every surface, with several AAA criteria as defaults. If a feature isn't usable by someone with a motor, visual, cognitive, or auditory difference, it isn't shipped.

WCAG 2.2 Conformance

Lumen conforms to WCAG 2.2 Level AA. Color contrast, keyboard access, focus visibility, and reduced-motion support are non-negotiable.

  • 1.4.3 Contrast (Minimum) — 4.5:1 text, 3:1 large
  • 1.4.11 Non-text Contrast — 3:1 UI components
  • 2.1.1 Keyboard — all functionality operable
  • 2.4.7 Focus Visible — visible focus indicator
  • 2.5.5 Target Size — 44×44px minimum
  • 3.3.2 Labels or Instructions — always present
Color Contrast

Every text-on-background combination in Lumen passes AA. Ink on warm off-white is 12.6:1; ink-3 on warm off-white is 5.8:1.

Ink on BG — 12.6:1
Ink 3 on BG — 5.8:1
White on Primary — 4.6:1
Ink 5 on BG — 2.4:1 (decorative only)
Keyboard Navigation

Every interactive element is reachable with Tab, in a logical order. Custom widgets follow WAI-ARIA Authoring Practices for arrow-key patterns (tabs, menus, comboboxes).

Focus States

A 2px accent focus ring with 2px offset is visible on every focusable element. Focus never disappears into the background. Use :focus-visible to differentiate keyboard from mouse.

ARIA

Use semantic HTML first; ARIA only when semantics are insufficient. aria-label on icon buttons, aria-expanded on collapsibles, aria-current on active nav items. Never use ARIA to override native behavior.

Touch Targets

Minimum 44×44px touch target on mobile. Spacing between adjacent targets is at least 8px to prevent mis-taps. Pointer events work for mouse, touch, and pen.

Reduced Motion

prefers-reduced-motion: reduce is respected everywhere. Animations collapse to instant transitions; the hero gradient stops drifting; spinners still spin (they're status, not decoration).

Screen Readers & VoiceOver

Tested with VoiceOver on macOS/iOS, NVDA on Windows, and TalkBack on Android. Hidden text via visually-hidden class, not display:none. Live regions announce dynamic updates.

Dynamic Type

Type scales use rem, so users' browser font-size preferences are respected. Layouts flex to accommodate 200% zoom without horizontal scroll.

High Contrast Mode

Tested in Windows High Contrast mode. Forced-colors media query adjusts borders and backgrounds. No information is conveyed by color alone.

Typography

Body text is 16px minimum. Line height 1.5+ for readability. Line length capped at 70-80 characters. No fully italic paragraphs (cognitive load).

Accessibility Checklist
Run this list before any component ships. If any item is unchecked, the component isn't done.
Semantic HTML used; no div soup for interactive elements
Color contrast passes AA (4.5:1 text, 3:1 UI)
Keyboard accessible — Tab, Enter, Escape, Arrow keys
Visible focus indicator on every focusable element
Touch targets ≥ 44×44px
Icon-only buttons have aria-label
Images have alt text (or alt="" if decorative)
Forms have visible labels and inline errors
Status changes announced via aria-live
Motion respects prefers-reduced-motion
Heading hierarchy is logical (no skipped levels)

A note on token discipline

Never hard-code a hex value, pixel size, or duration in a component. If you find yourself reaching for a value that doesn't exist as a token, propose a new one — don't reach for a magic number. The system stays coherent only as long as every decision routes through the token layer.
